# Design: Scalable Hardware Catalog via Build-Time Importer

**Date:** 2026-06-04
**Status:** Approved (pending spec review)
**Author:** Agent (brainstormed with user)

## Problem

`lib/hardware-catalog.ts` is a hand-curated static catalog (~80 GPUs, ~60 CPUs). Each
SKU is a verbose ~12-line TypeScript object literal, so the long tail of common gaming
hardware was never entered by hand. Concrete user-reported gaps:

- Intel **i5-9400 / i5-9400F** (Coffee Lake Refresh, 2019) — not present.
- "Many others are still missing" — e.g. Intel non-K mid-range across 8th–14th gen,
  i3 parts, AMD non-X / APU parts, and GPU gaps (RX 5500 XT, RX 6650/6750 variants,
  GTX 1080-class variants, etc.).

Adding hundreds more entries by hand bloats the file, causes merge churn, and invites
spec/perf errors. We need a way to scale coverage to hundreds of SKUs while preserving
the catalog's core properties: **pure, offline at runtime, deterministic, reviewable,
and all `perfIndex` values on one consistent gaming-weighted scale.**

## Goals

- Fill the genuinely common desktop-gaming hardware gaps (target **~120–180 new SKUs**,
  CPUs **and** GPUs).
- Make future coverage additions a data edit, not a code edit.
- Keep runtime behavior identical: no new network calls, no latency, `findHardwareByQuery`
  / `getHardwareEntry` just see a larger `ALL_ENTRIES`.
- Keep new entries on the **same `perfIndex` scale** as the curated ones.

## Non-Goals (v1)

- **No live runtime API.** Breaks the "pure, no runtime external calls" design and no
  good free benchmark API exists.
- **No live build-time scrape of PassMark.** Their list pages are scrape-only (ToS
  friction, brittle/large HTML). The committed dataset is the source of truth. A future
  `--refresh` flag could pull updated raw numbers, but it is out of scope here.
- No new component types (RAM/PSU/motherboard) beyond what the type already allows.
- No exhaustive OEM-only / mobile / Pentium-Celeron coverage (diminishing returns).

## Architecture

Three pieces, plus verification:

### 1. `seeds/hardware-catalog-dataset.json` — the data

Compact records (NOT verbose TS), one per SKU. The dataset is where coverage scales and
is the human-maintained source of truth for generated entries.

```json
[
  {
    "canonical": "Intel Core i5-9400F",
    "type": "cpu",
    "vendor": "Intel",
    "series": "Coffee Lake Refresh",
    "year": 2019,
    "cores": 6,
    "threads": 6,
    "tdpW": 65,
    "perfIndex": 31,
    "source": "PassMark CPU Mark (single-thread + gaming context), anchored to i5-9600K / i5-8400",
    "aliases": ["9400f", "i5 9400f", "i5-9400f"]
  }
]
```

Field schema (mirrors `HardwareCatalogEntry` in `lib/types.ts`, compacted):

- Required: `canonical`, `type` (`"cpu"|"gpu"`), `vendor`, `series`, `perfIndex`, `source`.
- CPU optionals: `year`, `cores`, `threads`, `has3DVCache`, `tdpW`, `notes`.
- GPU optionals: `year`, `vramGB`, `architecture`, `notes`.
- `aliases`: string[] — shorthands fed into the abbreviation map (lowercased).

`year` maps to `releaseYear`; `lastUpdated` is stamped by the importer at generation time.

### 2. `scripts/import-hardware-catalog.ts` — the importer

Reads the JSON, validates, and generates a typed catalog file. Run via
`npx tsx scripts/import-hardware-catalog.ts`.

**Validation (fails loudly, non-zero exit):**

- No duplicate `canonical` within the dataset.
- No `canonical` collision with the curated `GPU_CATALOG` / `CPU_CATALOG` (curated wins;
  a collision means the dataset entry is redundant and should be removed → hard error so
  the dataset stays clean).
- `perfIndex` within 0–115 (matches existing verify bound).
- Required fields present; `type` is `cpu` or `gpu`; `vendor` in the allowed union.
- Sane spec ranges (cores/threads > 0 and threads ≥ cores for CPUs; vramGB > 0 for GPUs).
- No duplicate alias keys (within dataset and against curated `BUILTIN_ABBREVIATIONS`);
  on collision the importer warns and **skips** that alias (curated/explicit wins) rather
  than emitting a conflicting map.

**Generation:** writes `lib/hardware-catalog-generated.ts` containing:

- `export const GENERATED_CATALOG: Record<string, HardwareCatalogEntry>` — all entries.
- `export const GENERATED_ABBREVIATIONS: Record<string, string>` — alias → canonical.
- A header banner: `// AUTO-GENERATED by scripts/import-hardware-catalog.ts — DO NOT EDIT.
  Edit seeds/hardware-catalog-dataset.json and re-run the importer.`
- Stable ordering (sorted by canonical) so re-runs produce minimal diffs.

### 3. Wiring in `lib/hardware-catalog.ts`

- `import { GENERATED_CATALOG, GENERATED_ABBREVIATIONS } from './hardware-catalog-generated';`
- `ALL_ENTRIES = [...GPU_CATALOG, ...CPU_CATALOG, ...Object.values(GENERATED_CATALOG)]`
  with a dedup pass so curated wins on canonical (defense in depth; the importer already
  forbids collisions).
- `resolveAbbreviation`: check `BUILTIN_ABBREVIATIONS` first, then `GENERATED_ABBREVIATIONS`
  (curated explicit shorthands win).
- `getHardwareCatalogStats` / `getCatalogVersionInfo` counts include generated entries.
- Bump `HARDWARE_CATALOG_VERSION` to reflect the importer addition.

### perfIndex — the one real judgment call

The existing `perfIndex` is **gaming-weighted and hand-tuned** (e.g. 9800X3D=96 ranks
above 9950X=82 despite far lower multicore). A pure linear normalization of raw PassMark
CPU Mark would misrank gaming performance. Therefore:

> Each dataset record carries its `perfIndex` **directly**, authored the same way the
> current file documents (single-thread + cache/core context, anchored to existing
> neighboring entries). The importer validates the range and ordering sanity but does
> **not** synthesize perfIndex from a formula.

This keeps generated entries on the identical scale as the curated ones, which is what
makes cross-entry comparison (the whole point of the app) valid.

## Data Flow

**Maintenance time:** edit `seeds/hardware-catalog-dataset.json` → run importer →
generated `.ts` is regenerated and committed.

**Runtime (unchanged, fully offline):** consumers call `findHardwareByQuery` /
`getHardwareEntry` / `getAllHardwareCatalog`, which read the enlarged in-memory
`ALL_ENTRIES`. When `NEXT_PUBLIC_USE_REAL_DATA=true`, live DB rows still override both
curated and generated via the existing `mergeCatalogs` mapper. Precedence:
**live DB > curated static > generated**.

## Generated File Handling

The generated `lib/hardware-catalog-generated.ts` is **committed to git** (not a build
artifact). Rationale: reviewable in PRs, no required build step for consumers/tests,
matches the "static source of truth" ethos. The importer is re-run only when the dataset
changes.

## Error Handling

- Importer validation failures → printed diagnostics + `process.exit(1)`, no partial write.
- Alias collisions → warn + skip the colliding alias, continue.
- Runtime: a missing/empty generated file degrades gracefully (catalog falls back to
  curated only); the wiring tolerates an empty `GENERATED_CATALOG`.

## Testing / Verification

- Extend `scripts/verify-hardware-catalog.ts`:
  - Assert generated entries are present in `getAllHardwareCatalog()`.
  - Existing dup / perfIndex / required-field checks now also cover generated entries.
  - Add search-resolution assertions for the reported gaps: `i5 9400`, `9400f`,
    `i5-9400`, plus a sample of other newly added SKUs resolve to the right canonical.
- Importer is deterministic: running it twice produces no diff.
- `npx tsc --noEmit` and eslint clean. **Do not** run `next build` against a live dev
  server (serves stale pages — see memory note); verify with tsc/eslint instead.

## File Touch List

- **New:** `seeds/hardware-catalog-dataset.json`
- **New:** `scripts/import-hardware-catalog.ts`
- **New (generated, committed):** `lib/hardware-catalog-generated.ts`

- **Edit:** `scripts/verify-hardware-catalog.ts` (assertions for generated coverage + gap searches)
